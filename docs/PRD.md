# SnapSOAP — Product Requirements Document (PRD)

**작성일**: 2026-03-13
**버전**: 1.0
**작성자**: SnapSOAP Team

---

## 1. 문제 정의

### 1.1 배경 — EMR 현장에서 목격한 문제

의사랑 EMR 개발 과정에서 수많은 의원·병원 현장을 관찰하며 반복적으로 목격한 장면이 있다.

> 의사가 환자와 대화하면서 한 손으로는 차트를 타이핑한다.
> 짧은 진료 시간(평균 5~7분) 안에 접수 → 문진 → 진찰 → 처방 → 기록까지 마쳐야 한다.
> 결국 차트는 "38.2 기침 3일 인후통 편도비대 편도염 Aug 3일"처럼 압축된 자유텍스트가 된다.

이렇게 쌓인 기록은 **해당 의사 본인만 해독 가능한 메모**가 되어버린다.
보험 청구 거절, 타 의료기관 전원 시 소통 불가, 의료분쟁 시 증거력 부재 — 이 모든 문제의 근원이다.

### 1.2 세 가지 현재 방식과 한계

| 방식 | 실제 사용 패턴 | 한계 |
|------|--------------|------|
| **자유텍스트** | 바쁜 의원의 90%+ 사용 | 검색 불가, 분석 불가, 품질 편차 극심 |
| **구조화 템플릿** | 대형 병원 중심, 일부 도입 | 입력 클릭 수 증가, 진료 흐름 단절, 의사 저항 |
| **음성→텍스트** | 극소수, 미국 중심 | 소음 환경(의원 진료실) 부적합, 후편집 부담, 비용 |

### 1.3 핵심 인사이트

> **"의사는 자유롭게 쓰고 싶고, 병원 시스템은 구조화된 데이터가 필요하다."**

이 딜레마를 AI로 해결한다.
입력 단계에서 의사의 워크플로우를 건드리지 않는다.
기록이 완성된 후, AI가 SOAP 구조로 재분류한다.

### 1.4 왜 SOAP인가

SOAP(Subjective · Objective · Assessment · Plan)은 1968년 Lawrence Weed가 제안한 이후 전 세계 의무기록 표준으로 자리잡았다. 한국 의료계에서도:
- 의과대학 임상실습 필수 교육 항목
- 의무기록사 국가시험 핵심 출제 영역
- 건강보험심사평가원(HIRA) 외래 기록 작성 권장 형식

SOAP으로 구조화하면 **보험 청구 근거**, **임상 품질 지표**, **타원 전원 요약서** 작성이 즉시 가능해진다.

## 2. 프로젝트 목표

### 2.1 제품 비전

의사가 평소처럼 자유텍스트로 진료 기록을 작성하면,
AI가 실시간으로 SOAP 포맷으로 자동 분류·구조화해주는 경량 웹 에디터.

**입력 방식을 바꾸지 않는다. 출력만 바꾼다.**

### 2.2 목표 사용자

#### 주 사용자: 1차 진료 의사 (내과·가정의학과·이비인후과 의원)
- 하루 50~100명 진료
- 진료 중 키보드로 차트 작성 병행
- 보험 청구를 위한 진단코드(KCD) 선택 부담 존재
- 구조화된 기록의 필요성은 알지만, 시간이 없음

#### 부 사용자
- **전공의**: 환자 정리 노트, 교수 컨퍼런스 준비
- **간호사/PA**: 처치 기록, 경과 기록 보조
- **EMR 벤더 기획자/개발자**: 자사 제품 도입 가능성 탐색

#### 간접 이해관계자
- **의무기록사**: 기록 질 관리 (미완성 차트 처리 부담 경감)
- **병원 경영진**: 보험 청구 거절률 감소, 의료 품질 지표 개선

### 2.3 성공 기준

| 지표 | 목표값 | 측정 방법 | 근거 |
|------|--------|-----------|------|
| SOAP 분류 정확도 | ≥ 85% | 전문의 리뷰 (10건 샘플) | 임상 현장 수용 최소 기준 |
| AI 응답 시간 | ≤ 5초 | API 응답 시간 측정 | 의사 인내 임계값 (현장 관찰) |
| 사용성 점수 (SUS) | ≥ 70점 | SUS 설문 (5명) | Industry standard "Good" |
| Task Completion Rate | ≥ 90% | 시나리오 테스트 | 클리닉 도입 실용 기준 |
| 의학 약어 보존율 | 100% | 자동 검증 | 원문 훼손 절대 불가 원칙 |

## 3. 기능 명세

### 3.1 MVP 기능 (반드시 구현)

---

#### F1. 자유텍스트 입력 패널

**설명**: 의사가 평소처럼 기록을 입력하는 좌측 영역

**동작 조건**:
- 최소 높이 400px, 자유롭게 리사이즈 가능
- 자동저장 없음 (의도적 — 민감 데이터 미저장 원칙)
- 글자 수 실시간 카운터
- Placeholder 예시 텍스트 제공

**Placeholder 예시**:
```
예) 38.2 기침 3일, 어제부터 인후통 심해짐.
편도 양측 발적, 삼출물 없음.
BP 120/80, HR 82. 편도염 의심.
Augmentin 625mg 3일 처방, 3일 후 재진.
```

**입력 지원 형식**:
- 한국어 자유텍스트 (주)
- 영어 또는 혼용 (SOAP, BP, HTN, DM, OA 등 의학 영문 약어 포함)
- 숫자+단위 표기 (38.2℃, 120/80mmHg, 75kg)

---

#### F2. AI SOAP 구조화 엔진

**설명**: 자유텍스트를 S/O/A/P 4개 섹션으로 자동 분류

**트리거**: "구조화" 버튼 클릭 (자동 실행 없음 — 의도적 제어)

**처리 흐름**:
```
[입력 텍스트 검증] → 빈 입력 시 안내 메시지
       ↓
[POST /api/structurize]
       ↓
[Claude API] 시스템 프롬프트 + 원본 텍스트
       ↓
[JSON 파싱] SoapResult 타입으로 변환
       ↓
[우측 패널 렌더링] 섹션별 표시
```

**출력 JSON 스펙**:
```json
{
  "subjective": ["38.2 기침 3일", "어제부터 인후통 심해짐"],
  "objective": ["편도 양측 발적, 삼출물 없음", "BP 120/80, HR 82"],
  "assessment": ["편도염 의심"],
  "plan": ["Augmentin 625mg 3일 처방", "3일 후 재진"],
  "unclassified": []
}
```

**분류 규칙**:
| 입력 패턴 | 분류 섹션 | 예시 |
|-----------|-----------|------|
| 환자 호소, 증상, 발병 기간 | Subjective | "기침 3일", "어지럽다고 함" |
| 활력징후, 신체 검진 소견 | Objective | "BP 120/80", "편도 발적" |
| 진단명, 임상 판단 | Assessment | "편도염", "HTN 조절 양호" |
| 처방, 처치, 추적 계획 | Plan | "Amoxicillin 처방", "1주 후 재진" |
| 분류 불가 | Unclassified | 맥락 없는 단어 등 |

**의학 약어 처리 원칙** (EMR 현장 경험 기반):
- HTN, DM, OA, CKD, GERD 등 한국 1차 진료 빈용 약어 — 원문 보존
- CC, HPI, ROS, PMH, FHx 등 SOAP 구조 관련 약어 — 분류 힌트로 활용
- BP, HR, RR, BT, SpO2 등 활력징후 약어 → Objective 자동 분류
- r/o (rule out), s/p (status post), c/o (complaint of) → 분류 맥락 해석

---

#### F3. 구조화 결과 패널

**설명**: 분류된 SOAP을 섹션별로 시각화하는 우측 영역

**UI 구성**:
- 각 섹션: 컬러 헤더 + 내용 리스트
  - S (Subjective): 파란 계열
  - O (Objective): 초록 계열
  - A (Assessment): 주황 계열
  - P (Plan): 보라 계열
  - Unclassified: 회색
- 내용 항목은 bullet 리스트로 표시
- 각 항목 클릭 시 인라인 편집 가능

**상태별 처리**:
| 상태 | 표시 |
|------|------|
| 로딩 중 | 스켈레톤 UI (섹션 형태 유지) |
| 성공 | 분류 결과 렌더링 |
| API 오류 | 재시도 버튼 + 원본 텍스트 보존 안내 |
| 빈 섹션 | 해당 섹션 흐리게 표시 (숨기지 않음) |
| Unclassified 존재 | 경고 배지로 주의 표시 |

---

### 3.2 부가 기능 (시간 허용 시 구현)

#### F4. 샘플 진료 기록

**한국 1차 진료 실제 케이스 기반** (모두 더미 데이터):

| 케이스 | 내용 요약 |
|--------|-----------|
| 급성 상기도감염 | 발열, 기침, 인후통 — 가장 빈도 높은 내과 케이스 |
| 고혈압 정기 방문 | HTN 관리, 복약 순응도 확인, 혈압 측정 |
| 요통 (급성) | 직장인 허리 통증, 근골격계 케이스 |
| 당뇨 합병증 모니터링 | DM + CKD 병발, 다중 진단 복잡 케이스 |

각 샘플은 실제 의사들이 쓰는 문체(축약, 혼용어, 비문)로 작성.

#### F5. 결과 복사 내보내기

- **마크다운 형식**: `## Subjective\n- 기침 3일...`
- **플레인텍스트 형식**: `[S] 기침 3일...`
- 클립보드 복사 후 Toast 알림

#### F6. 재구조화 (Re-run)

- 결과 편집 후 재구조화 버튼으로 다시 AI 분석 실행
- 이전 결과는 히스토리로 유지 (새로고침 전까지)

## 4. 사용자 플로우

### 4.1 주요 플로우 (Happy Path)

```
[메인 페이지 진입]
       │
       ▼
[좌측: 자유텍스트 입력]
  ┌────────────────────────────┐
  │ 직접 타이핑                 │
  │ 또는 샘플 데이터 선택 (F4)   │
  └────────────────────────────┘
       │
       ▼
["구조화" 버튼 클릭]
       │
       ▼
[로딩: 스켈레톤 UI 표시 (≤5초)]
       │
       ├── 성공 ──▶ [우측: SOAP 섹션 렌더링]
       │                    │
       │              ┌─────┴──────────────────┐
       │              │ S: 주관적 호소 (파란색)  │
       │              │ O: 객관적 소견 (초록색)  │
       │              │ A: 진단/평가 (주황색)    │
       │              │ P: 계획 (보라색)         │
       │              │ ?: 미분류 (회색, 있을 때)│
       │              └──────────────────────────┘
       │                    │
       │         ┌──────────┼──────────┐
       │         ▼          ▼          ▼
       │    [인라인 편집]  [복사]   [재구조화]
       │
       └── 실패 ──▶ [에러 Toast + 재시도 버튼]
                    (원본 텍스트 보존)
```

### 4.2 예외 플로우

| 상황 | 시스템 반응 |
|------|------------|
| 빈 텍스트로 구조화 시도 | 입력 영역 포커스 + 안내 Toast |
| API 키 미설정 | 설정 안내 페이지 또는 에러 메시지 |
| 네트워크 오프라인 | 오프라인 Toast, 입력 영역 유지 |
| API 응답 10초 초과 | 타임아웃 안내, 재시도 버튼 |
| JSON 파싱 실패 | 원본 텍스트 전체를 Unclassified로 표시 |
| 매우 긴 텍스트 (5000자+) | 글자 수 경고 + 처리 계속 허용 |

### 4.3 에러 처리 원칙

**원본 텍스트는 절대 소실되지 않는다.**

의료 현장에서 기록을 잃는 것은 치명적이다. AI 처리 중 오류가 발생하더라도 좌측 입력 패널의 원본 텍스트는 항상 유지된다.

## 5. 기술 요구사항

### 5.1 성능 기준

| 항목 | 목표 | 비고 |
|------|------|------|
| 초기 페이지 로드 | ≤ 2초 | Next.js SSR 활용 |
| AI 구조화 응답 | ≤ 5초 | Claude Sonnet 기준 |
| 타임아웃 처리 | 10초 | 자동 abort |
| 입력 렌더링 지연 | ≤ 16ms | 60fps 기준 |

### 5.2 브라우저 호환성

- **주 타겟**: Chrome 최신 버전 (한국 병원 PC 환경)
- **지원**: Edge, Firefox 최신 버전
- **반응형**: 데스크탑 우선 (1280px+), 태블릿 지원 (768px+)
- **미지원**: IE, 모바일 (의도적 제외)

### 5.3 보안 요구사항

- API 키는 서버사이드(Next.js API Route)에서만 사용 — 클라이언트 노출 절대 금지
- 입력 데이터를 서버에 저장하지 않음 (stateless API)
- HTTPS 전용 배포
- 환경변수 `.env.local` — git 커밋 금지

## 6. 비기능 요구사항

### 6.1 접근성
- 키보드 단독 조작 가능 (Tab, Enter, Esc)
- 스크린 리더 대응 ARIA 라벨
- 색상 외 시각적 구분 수단 병행 (아이콘, 텍스트 레이블)

### 6.2 국제화
- 한국어 입력 기본 (IME 입력 처리)
- 영어·한영 혼용 텍스트 처리
- 의학 약어(영문)와 한국어 진술 혼용 케이스 대응

### 6.3 유지보수성
- 프롬프트는 `src/lib/prompts.ts`에서 단일 관리
- 타입 정의는 `src/lib/types.ts`에서 단일 관리
- 컴포넌트는 단일 책임 원칙 적용

| KCD 진단코드 자동 매핑 | 후속 Phase로 보류 |
| 다중 사용자 동시 편집 | 복잡도 초과 |
| HIPAA/의료기기 소프트웨어 인증 | 프로토타입 단계 |

## 9. 향후 로드맵 (v2 가설)

SnapSOAP v1이 검증되면 탐색할 방향:

1. **KCD 코드 자동 추천**: Assessment 섹션에서 진단명 → ICD-10/KCD 코드 자동 매핑
2. **의사랑 EMR 플러그인**: 기존 EMR에 사이드패널로 임베딩
3. **진료과별 프롬프트 특화**: 내과·정형외과·소아과 등 진료과별 SOAP 패턴 최적화
4. **FHIR R4 출력**: 구조화된 결과를 FHIR 리소스(Condition, MedicationRequest)로 변환
5. **분류 피드백 루프**: 의사가 수정한 분류를 누적해 프롬프트 개선
